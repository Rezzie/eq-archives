Decaying Skeleton 		posted 12-17-2001 10:54 PM               
I am concerned about the potential for collection of usage statistics based on ip and time. OE is currently designed to inform slicer when it starts and stops on a computer. If this data were to be stolen or leaked out then Mythic could crosscheck the ips and access times and ban users. I realize I could block this with iptables or commenting out the code block every update, but would it be possible to add a switch to disable this feature?

Apart from this info leaking at it's source another danger is inherent in sending these stop/start flags over the internet.

Call me paranoid but what is to stop a pro-mythic sysadmin that admins public network space from creating an extra IDS rule to log these stop/start packets and mail them to mythic? A simple script could grep log files and send these to mythic automatically.

I think optimally a passive packet sniffer should be passive by default. I appreciate all the work that slicer and others have put into this and I can see where the usage data might be interesting and rewarding. OE sending out packets though does make me uncomfortable.

Innoruuk 		posted 12-18-2001 12:46 PM               
Sometimes the assumptions people take as facts amaze me.
There are no keepalive packets. There are no start packets. There are no stop packets.

Read oeUpdate.cpp

There's a single HTTP GET of the current version of OE and the checksum of the maps, which is a HTTP file residing on www.hackersquest.org. If you are paranoid (and stupid) enough to think I will sell log information to Mythic, you shouldn't be running OE in the first place, because OE contains BINARY components. I could have hidden anything in there, and you would never know.

If you still don't want to be visible in the hackersquest.org logs at all, first of all stop downloading anything from these pages, stop visiting these forums, and then type:

route add -host www.hackersquest.org reject

on your OE machine to make sure OE doesn't view that webpage.
