Many of us run very lightweight linux boxes that can't really handle openGL very well, if at all. I beg you to consider making a version that we can use. I see you spending a great deal of effort making a fantastic program, one that I think few of us will be able to use because of its complexity and because it's not well suited to the way DAoC handles dynamic objects.

The problem with what you're attempting to do is that as far as we've been told and can tell from the game, DAOC doesn't send character or NPC information unless the object is very close (perhaps 200 feet or less). This means that an overhead map isn't going to tell you very much, *unlike* ShowEQ, which was extremely useful becuse the entire zone was sent. Basically, if it's in the datastream, it's in visual range, and very nearby. The sole exception to this that I've seen is doors, which can be seen opening and closing from any distance within clipping.


For example, a text program is fine. All I really need is a program that pops up text messages when enemies are detected. Range and direction would be a bonus but aren't really all that important. This would make it difficult to be surprised by people over the next hill, or behind a wall. I expect you could write this in an hour

In fact, an even simpler version could run on the DAoC machine itself. It could either flash the keyboard lights, or insert snippets into the sound output when enemies are detected.

While your skill in programming is extremely admirable, (and it really is), a much simpler version could meet 99% of my needs on the battlefield.

So the three basic pieces of information I need in order of importance are:

A warning when enemies are detected nearby.

A rough count of them and their levels

Relative bearing to the targets

Anything else is going to be unnecessary because as soon as enemies close to this range, one is too busy fighting to worry about looking at an overhead map.

I'm a level 40 with substantial RVR experience, I'm willing to go over in more extensive detail the types of utilities that would help in combat.

Now maybe I'm wrong, maybe they send a huge boatload of messages out, and you'll figure that out. But even in that case I still can't use an open GL program, a simple 2d radar version would be preferrable to me.

Thank you for your time invested in this project.


Jerry

When I say "I'm using OpenGL to display the map", that doesn't mean I have 40 000 antialiased, trilinear filtered, multitextured, bump mapped and vertex shaded polygons, it means "I use OpenGL because it's a heck of a lot faster than X11 calls."
The drawing of the 2D map is about twice as fast using OpenGL that it would be with X11 drawing. If you have a slower machine and no 3D hardware, just turn OFF the lighting and antialiasing

No, I won't make a text version. I need a 2D version, so that's what I'm making

The program uses about 1% CPU on my P2-450, and uses the rest of the available cpu power to redraw the map as fast as it can, so I think most machines should be able to handle it just fine.
