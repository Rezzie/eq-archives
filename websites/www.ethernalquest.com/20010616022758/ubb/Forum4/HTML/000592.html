I can't believe I haven't heard about this from other FreeBSD users...

FreeBSD had some changes made to the threading system since 4.2-RELEASE which basically slow down ShowEQ significantly. It seems as if updates get paused for 10-20 seconds, and then it updates. It keeps going like this. So basically, ShowEQ doesn't show what's happening, it shows what happened 10-20 seconds ago.

I tried playing with nice values, setting real-time priority (using the rtprio program and ShowEQ's option to set the network thread to realtime priority), but they have no effect (other than to perhaps slow down the rest of the system). Top shows ShowEQ going idle for that 10-20 second period where it does nothing. The menus and stuff react instantly, but updates of EQ do not. If I con something, it also waits for that 10-20 second period to be over before it highlights it on the spawnlist.

As I said, it works fine on 4.2-RELEASE (updates are constant and nearly instantaneous), but it's slow even with a cvsup of today's -STABLE sources. It makes no difference if I'm running this on a Pentium-133 or an Athlon 750. In fact, the Pentium-133 running FreeBSD 4.2-R runs ShowEQ better than the Athlon-750 running the latest 4.2-STABLE.

I was wondering if anyone had any ideas on how to fix this. I don't know anything about programming with thread support, but.. perhaps there's some place in the ShowEQ source where thread priority can be changed?

I suppose I could switch to -CURRENT if I need to, but that's kind of a last resort.
