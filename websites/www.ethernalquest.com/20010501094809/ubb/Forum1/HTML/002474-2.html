Bone animation is usually done by giving each vertex several weights, which are then applied to nearby animated bones (matrices, transforms, whatever you want to call them) to get the final position, which is what we see on EQ characters.

Simpler bone animation simply attaches a set of polygons directly to one bone (matrix, transform, etc.), which is more or less what we see in doors. However, as this can be done by setting the weights to 1.0 on the given bone and 0.0 on others, this method can be done as a subset of the method above, so it's quite possible that the doors and such go through the character code.

Regardless of how hardcoded into a map something is, the complexity of animating it remains the same from the point of view of rendering polygons. You still have to have a transform for each different part of the tree that waves.

BTW, ever notice that pretty much everything on the map other than the ground comes out of the .chr file, not the .wld file? (Oh, that's right, you've never bothered to learn how those files are set up, have you?)
