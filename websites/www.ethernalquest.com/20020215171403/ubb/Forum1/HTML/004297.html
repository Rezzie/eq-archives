no, there is not a way.

patch works because it uses standard HTTP to download files, which socks handles with no problem.

EQ's protocol on the other hand is a propietary udp based protocol requiring specific ports to be allowed to connect. Read the firewall information page at everquest.com for more info.

Theoretically you could proxy an EQ connection, but you would have to write a lot of code. For one, an addon to socks wouldnt work, you would need to transaprently modify the protocol stream, and you would need to be pretty knowledgable with udp and ip in general.

The easiest way to do what you want is this.

somehow build up a ppp connection (possible tunneled over http if thats all you can use) to a host outside of your place of business. Route all your traffic over this ppp link, and setup appropriate software on the other end to masq your connection and then send the incoming packets back down the tunnel to your work box.

gollum, you are correct in your assesment of socks, but the problem with using that for eq is the general premise that socks works under. SOCKS, is not transparent, you need a client that knows how to use it, or some software that works in a transparent fashion that tunnels through SOCKS.

The basic dilemma, is EQ's protocol is an IP header + UDP header + propietary EQ header and data. You want to connect, but cannot directly.

To use SOCKS, you need to be able to tell your program, use SOCKS, and connect to port 1080 on box X, then do socks stuff to do your real connection.

This does not work with EQ as is. Once the game is initialized it starts sending udp data to a specific port of the login server, then to a specific port on the chat and then data to and from the the zone world and zone servers. There is no provisioning to connect to a socks server first, it just sends the packets outbound.

socks is all fine and well, but not for this application. If you really wanted to use socks \, you would need to write an app to transparently (to the client) intercept EQ's data, tunnel it through SOCKS, then on the other side, send it out as it should and handle return packets.

Why go through all this trouble when a tunnel and some routing rules (interface level stuff rather than needlessly tunneling through a higher level protocol). As stated in my earlier post, here is a setup that would work.

the boxes:
EQ box (inside work firewall)
tunnel box (inside work firewall)
(eq and tunnel box may be the same depending on software)
tunnelbox2 (outside firewall).

if your work firewall allows outbound connections at will, then do this:

build a TCP connection from tunnelbox to tunnelbox2. Now create a tunnel over this connection, ppp is a popular choice, and can be encrypted via ssh so work nazis dont see you playing EQ. Now, make your EQ box's default route tunnelbox, and make tunnelbox's default route the ppp link. Now ALL traffic from the eq box is going over this existing TCP connection. on tunnelbox2, masq the packets so they come back to tunnelbox2, and then forward the return packets back over the tunnel and to eq box.

if your work does not allow outgoing connections, it can still be done, but you will need to build your tunnel through thier socks servers if they have them. Note, you arent using socks directly, but only as a means to build your tunnel.

socks2http will only work for patch

patch uses http, the EQ game itself does not. for one, http is a protocol based on TCP, you connect to a webserver, issue a request and get a response back.

EQ's protocol is UDP, it is nothing like http, and tunneling them onto an http request is not going to work. The EQ servers (barring patch) do not respond to http requests, and in the normal course of the game never use a TCP connection.

even if you tell socks2http to connect to say, the world servers well known port, you have a TCP connection, the server wont accept it and wont talk back.

Zombie 		posted 11-16-2001 09:24 PM            
try this ~

pick a box as your internal routing point for the tunnel. From this point on, the box will be referred to as "internal". Pick a box outside work (at home?). This box is now called "external".

We'll assume internal and external are linux boxes because that makes this easier on me.

We'll also assume socks is required to get out of your work net because you cant stop talking about it.

go to internal:
rtelnet external

(this will telnet though socks to box external, you'll need to configure your SOCKS settings for rtelnet to work).

now, using the program pppd (read the man page, i dont use this prog) start a point to point connection over this telnet connection.

your ppp connection will get an interface name on both boxes, lets assume its ppp0 on both boxes.

now do this on box internal
setup a route for your local net and socks host if needed.
delete your default route
add a default route to ppp0
(alternativly, if you dont want to mess with the default route, you can add routes to all of EQ's nets to ppp0 instead of the previous two commands)
turn on ip forwarding
( echo "1" > /proc/sys/net/ipv4/ip_forward)

then on box external
setup a route to your work network over ppp0
setup SNAT on external (so packets come back to external rather than trying to get to your work box directly)
man iptables for that.

finally, on your EQ box, point your default route to box internal.

you should be able to play EQ now from work.
