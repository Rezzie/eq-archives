Hey Dirtface!

I've seen a number of assumptions in this series of posts about your install. How about consolidating your detail in one post for all to see.

1. OS and distribution
2. full details on NIC (version numbers drivers, model number, etc)
3. Network architecture
- what kind of router
- using switch or hub? manufacturer and model

A few other questions (which you might also take as suggestions) -
-have you carefully read the Ethernet Howto?
-is your NIC listed as supported
-have you found specific drivers for your NIC, if yes please state where (include URL if applicable)
-check out ifconfig (type man ifconfig from a Linux prompt) -- use it, love it, live by it.
-check out your /etc/rc.d/init.d directory. You'll find some great stuff in there to read up on (suggest you check network, smb, and anything else that looks interesting)
-check out the ifup and ifdown commands

The trick to getting a network card working is to first make sure you have drivers that cover the card. The latest distributions of Mandrake and RH seem to cover most cards. I used to have to DL and manually install tulip drivers for my cards (what a pain). With the latest packages, I don't have to do that any more. Next, make sure Linux recognizes your card. There's a simple way to check for recognized devices that I cannot seem to remember atm, but I'll post here if or when I do remember. You may have settings on your card to check. Finally, if you have drivers, Linux sees the card, the card is configured correctly (and passes diagnostics), you have a good cable connection the only thing left is to verify that Linux is starting your card. I use linuxconf to configure my cards as it's quick and easy, but it's not as manual as you might like for testing a card. For a "manual card load" use the ifup command then verify the card loaded correctly with ifconfig. If you have a good card, drivers, etc, and you have a compatible network mask you should minimally be able to ping other systems on your network.

Finally, the simplest way to verify you have a functional TCP/IP stack and card is to do a loopback ping. If you have a bad cable or almost any network problem, a loopback ping will tell you your system is ready to talk, but the network isn't. Simply type from a command prompt

ping 127.0.0.1

If you get responses, any problem is either network or cabling. In rare cases, you might have a HW problem with your card, but I think this would be very unusual.

Good luck!

OK Dirtface,

I'll say it again - read my post above and provide some detail. If you have to, cut-n-paste so you have my template to work from so you are sure to provide all the detail necessary.

There are a number of posts in this forum that are old, but still provide valid network infrastructure and architecture information. You don't have to be an expert to follow them.

Can you run SEQ? That question has a simple answer to start with. You must have a system to run Linux with SEQ and a separate system to run the EQ client. The Linux system does not have to be your gateway, although many have chosen to do just that because it is the simplest configuration. My personal preference is to have the Linux system as a peer TCP/IP client. I have run multiple EQ clients on my network and SEQ can successfully work with any of them. I've even switched using the "next IP" function, although that comes with its own specific challenges.

Summary - if you are saying you have a network that is DSL connected to your ISP with a router providing connectivity for your network, that is a workable scenario. If you then have peers on that network that are using TCP/IP (your Linux system and one or more EQ client systems) then it continues to be a workable scenario.

-ISP-DSL-Router-Hub(or switch)
-Linux with SEQ
-EQ Client
-EQ Client

There are two key requirements for this scenario. You must have a hub or switch that allows IP traffic to hit all nodes. That means it cannot be "discrete" or provide "bandwidth conservation" -- all systems see all packets on the network. Most switches do not allow for this, but some do. I'm running a 3Com 10/100 switch that is only discrete by speed meaning all the 10M systems see their specific traffic but not the 100M traffic and all the 100M systems see 100M traffic, but not 10M traffic. 2nd you must learn how to control your Linux system and keep it in PROMISCuous mode. It's very simple and just reading a few HOWTOs or doing a keyword search (on "PROMISC") will yield the results you need. If your Linux system is not running in promiscuous mode, IP traffic will not be visible to applications running on that Linux box regardless of your hub or switch.

If you are saying that you want to dual-boot your SEQ box with Windows using an EQ client, obviously when that system is booted to Windows you will not be able to run SEQ. Getting a small box to run Linux on is a very inexpensive proposition. You would get very good results from a Pentium or PentiumII 400MHz that has 128M of RAM. A system like this might run you under US$300 if you shop around. I have a friend that runs his SEQ on a PII 500 he picked up for $25 from a company in Atlanta that was selling old systems and writing them off (and yes, that $25 included kbd, mouse, and monitor).

Again, get that detail up if you need help. Cut-n-paste if you have to to be complete. Detail, detail, detail!!!! That's how people can trouble-shoot your difficulties. Less detail means more guess-work by people trying to assist.

Good luck!
