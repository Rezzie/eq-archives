Decaying Skeleton 		posted 04-01-2000 04:41 AM            
Just a warning to the development team: It's clear from my crashes (unreported) and others, that there are ways data coming from Verant can crash ShowEQ. The probability is these crashes are caused by buffer overflows, which may be related to trusting the packet data to fit "expected" guidelines, and therefore not doing proper checking.

Because may people use ShowEQ as root on their box, and because it's been shown time and again that a buffer overflow can allow arbitrary code to be run as the executing user, as long as ShowEQ is crashable, we are at risk.

For people concerned about this risk, you CAN protect yourself and continue to use ShowEQ without modifying the program -- IF you do not need promiscuous mode to access the data. Run ShowEQ as a special user that has no rights to do anything else on your machine.

To allow ShowEQ to be safe via code, open the raw socket, place it in promiscuous mode if requested, and drop all root privilages. Prior to dropping root, consider a chdir() to a benign location so no system commands can be executed.

Yeah, what Bellhead said...

for anyone interested, a temporary solution to the SEG faulting that seems to work for me is commenting out lines 530 and 531 in showeq/src/map.cpp

The lines look like

if (spawnItem[n])
delete spawnItem[n];

and should be changed to

// if (spawnItem[n])
// delete spawnItem[n];

This introduces other problems, such as spawns not being removed from the list after they're gone from the map... but at least it's not SEGfaulting.
