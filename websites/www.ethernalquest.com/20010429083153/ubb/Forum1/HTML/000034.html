Okay - this is for alla youse who wanna know whatcher looking at with EQInside.
First, the .pfs files are all sound files - simple .wavs, so you can play them with anything. I won't be dealing with them.

The .S3D files are the interesting ones...
First, (if you haven't already), get ACDSee to look at bmps.
Now, you will notice two primary extensions extracted by EQInside from the .S3D files:
.bmps (These are all the images that you see in EQ) and .wld (World files, these assemble the zone, and put the images where they belong.)
Skipping the .bmps for now (you can look at them on your own time - some very interesting things hidden in there...), let's move on to the ever-intriguing, yet apparently indecipherable .wld files.
First off, you might have noticed that there are three .S3D files for every zone (plus a bunch of extras that don't correspond to any zone). These 3 files (ZONE.s3d, ZONE_ chr.s3d and ZONE_obj.s3d) correspond to:
1) the basic zone itself (including the map, lighting and fog instructions and object locations)
The ZONE.s3d file (The "world" .s3d) has 3 .wld files in it:
a) ZONE.wld - this is the basic map of the world itself, where hills are, where exits are, and where each texture appears.
b) lights.wld - the dynamic lighting (and possibly fog?) instructions for that zone. (So that some areas are darker than others...)
c) objects.wld - I think that this is the instructions for where to place permanent non-structural objects (like doors and chairs and stuff)
2) the characters (NPCs) that can appear in that zone (note the severe limit on how many there are in each zone!)
The ZONE_chr.s3d file only has one "wld" file in it - that's the ZONE_chr.wld file, which apparently instructs which NPC uses what polygonal model to wrap their texture on.
3) the objects (such as chairs, fires, bookcases, etc.) that can appear in that zone.
The ZONE_obj.s3d file also only has the one ".wld" file - ZONE_obj.wld, which I would assume defines sizes and shapes of said objects.

We know the following information is controlled server-side (either by observation or process of elimination):
All basic character/world interactions (no peeking to find NPC scripts), character size (yes, actual physical size - haven't you seen "Giant Gnome Syndrome" before?) all character stats and attributes, all spell effects, etc.
What's not controlled server side? Textures and shapes for one. Also, believe it or not, emotes must be, because those people who tested emotes on the test server reported that the emotes stayed with the character when they went back to their "home server." Spell data is apparently partially stored locally, and partially stored on the server. Duration, for example, is strictly server-based, whereas most "effects messages" appear to be client-side.
Well, until someone builds a model-maker for us to play with, that seems about all we can dredge out of there.
