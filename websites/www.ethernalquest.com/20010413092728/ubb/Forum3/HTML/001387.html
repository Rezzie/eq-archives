Zombie 		posted 03-08-2001 12:55 PM            
Promiscuous mode means all packets that the network card sees get sent to the OS. In order to prevent excessive CPU load on a computer, the network cards normally check the address of the packet (the MAC address, not the IP address). If the packet is intended for a different computer, it ignores it. Promisc mode removes this feature and allows all packets, regardless of destination, to be viewed.

What class your network is makes no different, wheather it be a private class A (as is usually the case with ISP's granting a 10.x.x.x addy), or a public class C addy, or an addy from the reserved block (forget the range). Heck, you don't even need an IP address to sniff a network, depending on protocol (I think you might need the IP part for TCP/IP). Some hardware monitors don't even have a MAC address, but they are a special case.

Your description of your network confuses me a little (never seen the acronym PAT before. Do you mean NAT perhaps?), but as for the promisc mode being detectable...

There are ways to detect a promisc card, and there are ways to circumvent these methods. A basic method involves a bradcast and checking it against known IP addys, because promisc cards supposedly don't respond to broadcasts, but this is easily circumvented by disabling IP on your sniffer (not sure how this will affect sniffing, but from what I remember of networks it should work fine). There is a similar technique that uses known MAC addys (every NIC has one), but both methods require pre-knowledge of the addys. VI doesn't know my IP and MAC address, or the other computer's IP and MAC address, or the firewall's IP and MAC address. They would have to sniff packets to get these addys...

I personally doubt they would sniff the packets (I know 2 guides and a sr guide and haven't heard anything), since it is playing with fire. Granted, they could claim that the EULA precludes privacy and grants them permission to scan your network, but EULAs don't hold up too well in court now do they? Verant isn't as stupid as the people here would seem to like to believe.

But, for the ultra-paranoid, I will give you a safe network setup.

<WAN access point (cable/dsl, gateway comp, etc...>
|
<Hub1>---<SEQ computer>
|
<Switch> or some other kind of bridging device (bridge, firewall, router, etc...)
|
<EQ computer>, or <Hub2>---<EQ computer>, or <internal network>

This forces the packets to pass the SEQ computer while isolating the SEQ comp from the EQ comp.
It is impossible to detect the promisc card on <SEQ computer> because broadcast packets are not forwarded across the switch.
Even if it were possible to detect this NIC, there is NO WAY for anyone to distinguish the location of the promisc card (Is it set up like my diagram, or is it in the ISP's domain, which is not uncommon, or is it some backbone router being monitored for hackers, or is it an internal sniffer making sure all of VI's servers are sending what they should be sending, you get the idea?)

I also saw a post in this thread about cable modems. The DSL providers abuse the term "shared" when they describe cable providers and make it seem like all of your neighbors can see YOUR network traffic. (The other misleading details are not subject of this thread, board, or even forum.) Truth is, in the manner that your connection is "shared," so is EVERY other ISP in existence. Including dialups and DSL. The network layout looks different physically and logically, but for all practicality there is no difference. You can't sniff your neigbor's EQ session, whether you are both on the same dial-up, the same DSL, the same cable, or two completely different providers.

For the doubtful, the cable-modem is actually an edge router, which means that only packets bound for your home network will enter your home network. This is to reduce network load, and without it the Internet would not be possible. Packets dropped by the modem include all broadcast packets, packets not meant to cross the modem (ie. between your own computers if you're having a LAN party) and usually any packet destined for a private IP address (unless your ISP uses a network of private addys to save money).

Computers with a modem using internet sharing software function as an edge router the same way that the cable modem does, as do all DSL and ISDN modems.

Decaying Skeleton 		posted 03-08-2001 09:46 PM         
FYI - "PAT" is the acronym for "Port Address Translation" sometimes referred to as "Nat overloading". As I understand it, true NAT uses a pool of registered IP addresses on the outside interface of the translation device. When a client on the private network initiates a session to host on the public side of the device, the device assigns a public IP address to the requesting host and forwards the request out with the source address from the pool. The public host responds to the source address of the pool, and the translation device looks the destination device up in its translation tables and forwards the response to the original requestor on the private net. PAT works similarly, but has 1 public address on its outside interface only. It keeps track of all the different requests via TCP & UDP ports associated with the single, public address on its outside interface, instead of using a pool of public IP addresses. I hope that sounded coherent
