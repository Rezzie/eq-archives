I'm interested in editing my .opt files. The socials data is mostly obvious, 9 characters for button title followed by five 60 character sections for each socails line of text. This seems simple at first, until you notice "old" social data in there that has been overwritten with newer data (given the newer data is fewer characters than the older data). For example if you had a social that used to say

/g pulling, be ready

and then later editied it to say

/g inc %t

The social data for that line in the .opt file will appear like this:

/g inc %tg, be ready

Note, the "new" data overwrites the "old" data, however the old data is not erased. Also note I may be forgetting a space added to the end of the new data (can't recall if it does this or not), in which case it would actually look like this:

/g inc %t , be ready

Anyway - there is a large block of data prior to the socials portion of the file that I cannot decipher. I assume this contains some sort of line-length information that properly truncates each social line so the "old" line data is not brought into the game. I would also assume that the color of the button titles are saved in this section. However, this mystery section is quite lengthy, so some other data is presumeably being stored here as well. For all I know the game settings (such as graphical, sound, mouse, chat filter, etc) are kept in this block as well.

I'm wondering if anyone knows exactly how the data is saved, what the format is.

Yea, I thought this is how it would work too. But the whole socials area is just 1 big string. I opened the file up in my byte editor to check it out, but the character that seperates the old and new data is 32 - ASCII 'space' (32 decimal / 22 Hex). And this character is repeated many times throughout the valid data that EQ would want to read into the game, so it can't be used as a terminator. The entire socials area is essentially one big string, with no non-printable ascii in it at all.

Here's another funny thing - because no nulls are being used, each of the hotkey sections is totally uniform. Exactly 9 characters are allowed for the button title section, and exactly 60 characters are allowed for each actual line of the social. Where a lack of 9 or 60 characters occurs, the space character buffers the remaining characters until a new line or new button title occurs. So in every single social you have, EQ is always trimming off the excess space characters at the end of it (unless you have one that's exactly 9 characters in the button title and 60 characters in the text line).

So this is what's confusing me. Blank socials that you would presume have no data at all, actually contain 9 spaces followed by 300 spaces for the 5 "empty" lines. This lead me to assume that the block of data prior to the socials actually contained some sort of string-terminator data so that EQ would know when to stop reading a line of social data. Why they would do it that way, I don't know, but I can't think of any other way they may have constructed it. When I delved into that big chunk of nonsense, I couldn't make heads or tails of it, so... I'm basically stumped as to how this works.
