I think placing a recast timer on each slot is the right way to go. This should only apply to the 16 equipped slots and the 8 top level inventory slots. No more activating items from inside of bags. If these timers were done as simple countdown timers stored as a single byte for each slot and decremented on every tick, you could get any recast delay you wanted from 0 to 25.5 minutes (in six second increments) with only 24 bytes of storage per character. Every tick, the timer is decremented and if the timer is 0, the item can be activated.

Now for the real trick to prevent exploits due to trading/swapping items. The timer has to be reset when the item is placed into the slot. This would correspond exactly to the way memorized spells work. They are not available immediately after memorizing; you must initially wait for the recast time. If you have an item with a very long recast delay, you would have to prepare for it by equipping it in advance.

There are still some flaws with this idea to be worked out. If the timers are only kept in server memory when the character is active, zoning or camping would necessitate resetting all timers. This could be bad for items with long recast delays, or even for shorter delays if the item must be used quickly upon zoning (Mask of Deception when zoning into a KoS area). Therefore, these timers would necessarily have to be permanently stored in the character database.

Ideas that require changes to the character database are usually rejected, but what if we just altered the meaning of some bit of existing character data?

Now for some speculation about the internals of EQ itself. There must be an ID in the character database for each of the slots to hold which item is in that slot. I would expect that that number is 32 bits (as 16 bits wouldn't allow enough items) allowing for 4 billion+ different items. Are there that many items in the game? I don't think so. Are there more than 16 million or so? Maybe, but again, I don't think so. If not, then we could change the 32 bits used to store an item ID to only use 24 bits for the item ID and use the remaining 8 bits for the slot timer. To avoid a massive database change during the patch when implemented, allow everyone to get the first use of an equipped activated item free (without waiting for the recast time). I don't think that would be overly unbalancing, and would certainly negate itself after 25.5 minutes of play anyway.

What do you all think of my idea?

<sigh> Programmers of the world, UNITE! <grin> Long post, sorry.

Here's the architecture as far as I see it:

DB - The DB is more then likely going to have to be, like all DBs, normalized to heck and back (unless they are looking at access times, in which case, it may be denormalized to heck and back just to get the query performance. Caching should fix that, shouldn't it?).

Handling static information is pretty easy. Store the general item characteristics in a table and FK back to the character table (with maybe some sort of either multi-join table in between or denormalized coding for speed, not sure which). The client datastructure need only store the key and maybe summary info about it and link back to the DB (since we query an item much less then we use it).

The dynamic stuff is the key though and is stored,um, where? Two solutions to that: either store it on the server and update the client or store it on the client and update the server.

If it's stored on the client (data structure numerically limited and probably dynamically allocated from that Global file we hear so much about) then we can add whatever mechanisms are needed to calculate recast (since no-one other then the player needs to know how long until the next recast) on the client and just make a local call to figure out what to do next. This is almost certainly stored as a decremented tick count: 0 meaning you can recast (why bother with the DT overhead?). Certainly, this is part of that global datastructure (or maybe spell datastructure. Procs are, after all, just spells right?). Each item has a field that contains the spell PK of its proc and the proc is handled like any other spell. That brings me to another problem: do spells have a recast time (I play a Rogue so wouldn't know)? If they don't, maybe they should. *That* would be a wise architecture change methinks.

Now, here's where it gets interesting...if items have a spell slot and that spell slot goes to the spell datastructure (which already holds all the spell information) then the only issue is if spell recast is different from proc recast (which may be the case). Solve that by simply adding another spell to the spell database. Procs, most certainly, calculate spell casting in a different way (mana usage, comes to mind) so there has to be another bit of code that executes only for procs (or negates certain spell casting tests). Good design would put the two in the same function but with different parameters (so you can reuse the code, right?). If they have either 4th normal form or relatively well normalized data in the DB then adding or deleting "stuff" (spells, armour, equipment, characters, whatever...) should be straightforward: just add another record to the appropriate table.

If everything is stored on the server, then, again, we just calculate the tick count as with all spells and carry on. The issue becomes communicating another number back with the regular update traffic.

No, I think there is something more going on here.

So far as I can see it, there are three issues preventing the changes (none of which are architectural).

1) Any changes to the global file will require all of the file to be retransmitted (poor file design, IMHO) which, at however many megs, is a pain both on the network and on the client. Same with any code changes to the client.

2) Datastructure communication. By that I mean that obviously the datastructure and the DB tables have to be synched up to load information. Adding a field to the DB means adding a field to the datastructure which just brings us back to problem 1.

'Course, all of this assumes that Procs are spells. If not, why not? That would be the architectural change I'd make, lickity-split.

The sticking point could be that spells don't have recast times and they are now trying to make some spells (ie// those cast from items) have additional attributes (like recast) that they didn't architect in the first place. To change that, they have to change the datastructure (for spells to include both the total recast tick time and the current ticks left till recastable) and the DB (add the values per spell). The more I think about it, the more I think that spells don't have a recast time and, therefor, Procs are not spells glued to items. Hmmmm, sticky problem.

Maybe make the first couple of bytes of the name actually encode binary values that correspond to the recast time and time to recast respectively? Nice hack, but I wonder if there isn't a better way? Anyone want to speculate on the total EQ Architecture? <grin>
