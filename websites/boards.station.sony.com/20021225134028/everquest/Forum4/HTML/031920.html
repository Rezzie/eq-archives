"Where do you get the 256 number from?"

As I (briefly) explained, I was making a guess that is a vast overrepresentation of efficient ways to do this.

In theory, it's possible to represent a hierarchical database (which is what your bank space is) that only contains references to an external database (the database of all possible items) with nodes (a bank slot is a node) consisting of a quantity field and a reference to the external database. Since the quantity field can't represent a quantity over 20, this only requires one byte. An external database reference, no matter what the type, can be represented in four bytes (a 32-bit pointer) or less. That's five bytes. Since each node (bank slot) can have a subnode (container) with up to ten slots, this means that the maximum needed, given a sufficiently efficient storage mechanism, is 55 bytes per bank slot - 5 for the slot itself and 50 for the potential container slots.

So 256 is a huge overestimate of what's needed - but, since I do not work for SOE, I don't know what's actually used, so I multiplied that by a factor for memory-ineffeciency.

In other words, it was a SWAG. (Scientific Wild-Arsed Guess)
