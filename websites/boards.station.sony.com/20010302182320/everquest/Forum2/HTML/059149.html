This proposal came up dozens of times before with no official confirmation that it's been worked on, although some blue name once said they are looking into it. It is such a versatile and useful tool that I believe it should be pushed near the front of the priority list.

PROPOSAL: /guildmotd <insert message here>

The full version of this command will only be available to guild leaders.

Executing it with a string parameter stores the string as the guild 'message of the day' (up to /tell-length long) in the guilds database on the server.

When guild members log on, the text is fetched from the guild database and played back to them in yellow text, right after the server MOTD. This text cannot be blocked, is played back once on login, and the player does not need to execute a command to see it, although /guildmotd without a parameter will play back the message for any guild member at any time.

Examples of how this command is incredibly valuable to guild life: imagine that you just logged on, and you see one of these messages in yellow text. Without the /guildmotd command, you have to rely on website boards that people don't read or don't remember (or pretend to not remember / not seen) etc etc etc.

"Tindi'Losi - we are raiding in Chardok on Sunday, 3pm EST, meet outside Chardok. Bring Fire resist gear!"

"All guild members are invited to the wedding of ___ and ___ to be held on Sunday, 3pm EST, at the Temple of Marr in Freeport."

"Guild members, we have a Plane of Growth raid on Sunday, 3pm EST! Be there!!"

"Please welcome Beavis and Butthead, both level 1 druids, to the guild!"

"Guild exp groups tonight in Karnors and Sebilis - all welcome, levels 50+ "

"SCAM WARNING - do not trust anyone claiming to be an alt of a guildmate and asking for equipment or account info. Make them prove it!!"

"Guild meeting on Sunday the 18th, 8pm EST, chat server. Usual password!"

Seen a bluename response on this one before. It's apparently something they backburnered because it's a pain to code (without sucking up lots of resources). Apparently the way the guilds are coded is pretty inflexible. I'm guessing it's something like this:

Table: Guilds
Fields: Guild ID number, Guild Name

Table: Characters
Fields: among 50,000 other fields (ok, that's an exaggeration) Guild ID number

So first, they would have to add to the guild table. Adding a field to a table is changing database structure, and is not done lightly. That would have to be a text field, and a long one.

Then, after they load your guild tag, but before they drop you in the game, they have to load your guild motd and display it, if there is one. If there isn't, they have to not display it (they actually have to code that in, remember... computers are dumb).

Finally, they probably have to give it an expiration date, or better yet, a start date and expiration date. That's two more fields in the table, and then we need some kind of interface to set them.

A great idea, but not so easy... hopefully, they'll get done with all the balance and stuff and attack the guild structure next.
